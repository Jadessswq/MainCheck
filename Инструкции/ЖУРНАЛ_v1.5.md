**Железо:** Broadwell Xeon E5-2690 v4, 2 сокета × 16 ядер, 31 ГБ RAM, две NUMA-ноды по ~15.5 ГБ каждая. **Нет AVX-512** (она появилась только со Skylake Gold).

**Статус до сессии (теги v1.4.3):**

- На warm-запросе — **~100 с** ответ

**Запросы пользователя:**
- «Медленно, оптимизируйте таймауты» → на самом деле проблема была не в таймаутах, а в размере модели под слабый CPU.

## 4. Проблема NUMA: qwen3:30b OOM

Первая попытка запустить `qwen3:30b-a3b-instruct-2507` упала:

```
апр 29 16:49:42 ext numactl[...] architecture=qwen3moe file_type=Q4_K_M
апр 29 16:50:10 ext numactl[...] status="llm server error"
апр 29 16:50:10 ext numactl[...] error="llama runner process has terminated: %!w(<nil>)"
```

**Диагноз:** наш systemd-override для Ollama использовал `numactl --cpunodebind=0 --membind=0`. `--membind=0` = «аллоцировать память ТОЛЬКО из NUMA-ноды 0». На двухсокетной системе 31 ГБ RAM делится поровну → в node 0 доступно ~15.5 ГБ. qwen3:30b-a3b Q4_K_M весит 17.3 ГБ (+KV-кэш ещё ~400 МБ) → **в одну ноду не влезает**, ядро убивает процесс.

Подтверждение нашлось в `free -h` во время запуска: общая память свободна (30 ГБ), но это суммарно по двум нодам. Ollama с `mmap=false` аллоцирует всё сразу → падает до того, как ядро начинает swap'ать.

**Workaround:** временно заменить `--membind=0` на `--interleave=all`:

```bash
sudo mkdir -p /etc/systemd/system/ollama.service.d
cat | sudo tee /etc/systemd/system/ollama.service.d/numa-interleave.conf <<'EOF'
[Service]
ExecStart=
ExecStart=/usr/bin/numactl --cpunodebind=0 --interleave=all /usr/local/bin/ollama serve
EOF
sudo systemctl daemon-reload && sudo systemctl restart ollama
```

После этого qwen3:30b запустился. Первый голый тест (без RAG, без жёсткого формата) показал **46.7 с** на /api/chat — это выглядело как победа. Но:

1. Модель **внесла НОВУЮ ошибку**: «приказа директора» → «приказа директором» (граматически неверно).
2. Объединила два самостоятельных предложения в одно, потеряла смысл.

То есть тест без полного промпта вводил в заблуждение. Нужен был полноценный прогон с RAG и строгим форматом.

## 5. Финальный бенчмарк bench7.sh

Прогоняем 5 моделей (включая прежний дефолт и baseline) через полный промпт с RAG-контекстом и требованием формата `===CORRECTED===/===CHANGES===/===END===`.

### Скрипт bench7.sh (сокращённо)

```bash
MODELS=(
  "qwen3:30b-a3b-instruct-2507-q4_K_M"
  "forzer/GigaChat3-10B-A1.8B"
  "t-tech/T-lite-it-2.1:q4_K_M"
  "qwen3:14b"
  "qwen2.5:14b"
)
for m in "${MODELS[@]}"; do
  echo "===== $m ====="
  # preload
  t1=$(date +%s)
  curl -s http://localhost:11434/api/generate \
    -d "{\"model\":\"$m\",\"prompt\":\"\",\"keep_alive\":\"5m\",\"options\":{\"num_thread\":16}}" > /dev/null
  t2=$(date +%s); echo ">> preload: $((t2-t1))s"
  # warm bench
  t3=$(date +%s)
  curl -s -X POST http://localhost:11434/api/chat --max-time 600 \
    -H 'Content-Type: application/json' \
    -d "$(jq -n --arg m "$m" --rawfile ctx /tmp/rag_ctx.txt --rawfile text /tmp/bad_text.txt '{
      model: $m,
      messages: [
        {role:"system", content:"Ты редактор русских официально-деловых текстов. Ответь ТОЛЬКО в формате ===CORRECTED===/===CHANGES===/===END==="},
        {role:"user", content: ($ctx + "\n\nИСПРАВЬ:\n" + $text)}
      ],
      stream:false, keep_alive:"5m",
      options:{num_thread:16, num_ctx:4096, num_predict:1024, temperature:0.1}
    }')" | python3 -c "import sys,json;d=json.load(sys.stdin);print(d.get('message',{}).get('content',''))"
  t4=$(date +%s); echo ">> bench: $((t4-t3))s"
done
```

### Результаты

| Модель | Preload | Warm | 4/4 правок | Формат | Особенности |
|---|---|---|---|---|---|
| qwen3:30b-a3b-instruct-2507 | 19s | **119s** | ✓ + новый баг | ❌ цикл в CHANGES | «не подходит… не подходит… не подходит…» |
| forzer/GigaChat3-10B-A1.8B | 15s | **30s** | ✓ | ❌ дублирует текст | копирует справку из RAG как правки |
| **t-tech/T-lite-it-2.1:q4_K_M** | **10s** | **44s** | ✓ | ✓ чистый, 2/5 идемпотентных | **победитель** |
| qwen3:14b | 16s | 215s | ✓ | ✓ чистый | слишком медленно |
| qwen2.5:14b (прежний дефолт) | 42s | 86s | ✓ | ✓ чистый | baseline |

### Почему qwen3:30b оказался медленнее 14B

Парадокс: MoE с 3.3B active параметров должен быть **быстрее** dense 14B. На деле — **медленнее в 1.5×**. Причина — наш workaround `--interleave=all`: половина обращений к памяти идёт через cross-socket QPI-bus, а это x2 latency. На CPU инференс **memory-bound**, не compute-bound, поэтому каждый лишний наносекундный hit в память дороже, чем экономия на числе параметров.

Плюс к этому — reasoning/instruction-tuning в qwen3:30b-2507 сломал формат: модель не остановилась на конце CHANGES, ушла в цикл «не подходит/правильно: …/не согласуется». Post-processor не спас бы — ответ обрывался по `num_predict=1024` посреди рассуждения.

### Почему GigaChat3 не подошёл несмотря на 30 с

GigaChat3-10B-A1.8B (MoE 10B / 1.8B active) отвечает за 30 с, но вместо `===CORRECTED===` сначала выдаёт текст markdown'ом (`УОПМ ГУСБ Росгвардии… **согласно**…`), потом дублирует его же в `===CORRECTED===` блоке, а в `===CHANGES===` копирует куски RAG-контекста как «правки». То есть парсер на сервере просто не вытащит структурированные правки — выдаст «неверный формат» → клиент не покажет Track Changes.

Теоретически лечится few-shot примерами в промпте, но это отдельный эксперимент. T-lite работает «из коробки» — выбираем его.

## 6. Выбор T-lite-it-2.1

### Почему именно `:2.1`, а не `:1.0`

T-lite v1.0 (середина 2024) был base-моделью без instruction-tuning. T-Bank сам писал, что в production её ставить нельзя. V2.1 (декабрь 2025) — первая версия, основанная на **Qwen3-8B** с дообучением на русском официально-деловом стиле. Именно её T-Bank рекомендует как production.

### Почему 8B обгоняет 14B при том же качестве

1. **Специализация.** T-lite дообучена именно на русских деловых документах. qwen2.5:14b — general-purpose, русский в нём — один из десятков языков. На узкой задаче tuned-8B сравнима с general-14B.
2. **Размер.** 5 ГБ Q4_K_M вместо 9 ГБ → **меньше cache misses**, **меньше memory bandwidth** (ключевой узкий горлышко CPU-инференса). ×1.8 быстрее по времени warm.
3. **Помещается на одной NUMA-ноде.** 5 ГБ ≪ 15.5 ГБ → можно вернуть `--membind=0` (без cross-socket трафика). qwen2.5:14b тоже влезает, но с меньшим запасом под другие процессы.

### Недостаток: идемпотентные пункты в CHANGES

T-lite из 5 пунктов `===CHANGES===` иногда выдаёт 2 вида ««согласно распоряжению» — исправлено на «согласно распоряжению» (дательный падеж соблюдён)». То есть модель перечисляет правила грамматики, применённые к тексту, даже если по этому конкретному пункту правки не было. На Track Changes это превращается в пустые метки «правка: заменить X на X» — визуальный шум.

**Решение:** post-processor `_drop_idempotent_changes()` в `main.py`, фильтрует такие пункты регэкспом.

```python
# Угловые/типографские кавычки, встречающиеся в ===CHANGES===. Одиночные
# ' и ` не включаем — они ложно срабатывают на апостроф/транслитерацию.
_QUOTE_CHARS = "«»\"“”‘’‚‛„"
# Разделитель между «было» и «стало». Допускаем стрелки (→, ->), тире
# (—, –, -) и текстовые связки, включая обороты «заменено/исправлено на».
_CHANGE_PAIR_RE = re.compile(
    rf"[{_QUOTE_CHARS}]([^{_QUOTE_CHARS}]+)[{_QUOTE_CHARS}]"
    rf"[^{_QUOTE_CHARS}]*?"
    rf"[{_QUOTE_CHARS}]([^{_QUOTE_CHARS}]+)[{_QUOTE_CHARS}]",
    re.IGNORECASE,
)

def _drop_idempotent_changes(text: str) -> str:
    """Удаляет из блока ===CHANGES=== пункты вида «X → X»."""
    if "===CHANGES===" not in text or "===END===" not in text:
        return text
    try:
        before, rest = text.split("===CHANGES===", 1)
        changes_block, tail = rest.split("===END===", 1)
    except ValueError:
        return text

    kept: list[str] = []
    for raw_line in changes_block.splitlines():
        line = raw_line.rstrip()
        if not line.strip():
            kept.append(line)
            continue
        m = _CHANGE_PAIR_RE.search(line)
        # Сравниваем БЕЗ .lower(): «Приказа» → «приказа» — валидная
        # орфографическая правка регистра, такие сохраняем.
        if m and m.group(1).strip() == m.group(2).strip():
            logger.debug("Фильтрую идемпотентный пункт: %s", line.strip())
            continue
        kept.append(line)

    non_empty = [ln for ln in kept if re.search(r"\w", ln)]
    has_real_item = any(re.match(r"\s*\d+\.\s*\S", ln) for ln in non_empty)
    if not has_real_item:
        kept = ["", "1. Ошибок не найдено. Текст соответствует нормам.", ""]

    new_changes = "\n".join(kept).rstrip() + "\n"
    return f"{before}===CHANGES===\n{new_changes.lstrip()}===END==={tail}"

# wired в пайплайн:
# return _strip_thinking(raw)              # было
return _drop_idempotent_changes(_strip_thinking(raw))   # стало
```

## 8. Devin Review: баг с регистром

После создания PR Devin Review отметил валидный баг:

> **Case-insensitive comparison in `_drop_idempotent_changes` incorrectly filters legitimate case-correction changes**
>
> Filter at `Сервер/local/main.py:220` uses `.lower()` when comparing "before" and "after". This means any change that only differs in capitalization will be silently dropped. For example, `«на основании Приказа» → «на основании приказа»` is a real correction (word is not a proper noun), but both sides lowercase to identical strings.

### Фикс

```diff
 m = _CHANGE_PAIR_RE.search(line)
-if m and m.group(1).strip().lower() == m.group(2).strip().lower():
+# Сравниваем БЕЗ .lower(): «Приказа» → «приказа» — это валидная
+# орфографическая правка регистра (имя собственное vs нарицательное),
+# такие пункты сохраняем. Идемпотентный пункт — это когда до и после
+# совпадают побуквенно.
+if m and m.group(1).strip() == m.group(2).strip():
     logger.debug("Фильтрую идемпотентный пункт: %s", line.strip())
     continue
```

Плюс регрессионный тест:

```python
def test_local_case_correction_is_preserved(local_module):
    """Правка только регистра (Приказа → приказа) — валидная орфографическая,
    НЕ должна считаться идемпотентной."""
    raw = (
        "===CORRECTED===\n"
        "на основании приказа...\n"
        "===CHANGES===\n"
        "1. «Приказа» → «приказа» (слово не является именем собственным)\n"
        "2. «округе» → «округе» (ложная правка — дублирует текст)\n"
        "===END==="
    )
    out = local_module._drop_idempotent_changes(raw)
    assert "«Приказа» → «приказа»" in out  # сохранено
    assert "«округе» → «округе»" not in out  # отфильтровано
    assert "Ошибок не найдено" not in out
```

**Коммит:** 20c481c. PR #9 смёржен после этого фикса.

| **Sandwich Runtime** | Раздельная оптимизация prefill/decode | ⚠️ Пока research, нет production-реализации для open-source моделей на CPU. Ожидаем в vLLM/SGLang через 3–6 месяцев. |

### Неприменимо

| Идея | Что | Оценка |
|---|---|---|
| **T-SAR (1.58-bit ternary)** | Тернарное квантование, 5.6–24.5× на CPU | ❌ Работает только на моделях, обученных в 1.58-bit с нуля (BitNet-style). Просто «сконвертировать» T-lite нельзя — качество упадёт до мусора. Русских деловых BitNet-моделей пока нет. Ждём 1–2 года. |
| **SAIL (Processing-in-Memory)** | PIM-чипы (Samsung HBM-PIM, SK Hynix AiM), интеграция вычислений в память | ❌ Требует специальной памяти, которой в обычных серверах нет и не будет без смены платформы. Не в этом десятилетии. |

### Итоговая приоритизация

| № | Идея | Выгода | Стоимость | Статус |
|---|---|---|---|---|
| 1 | Qolda 4B bench | ~2× скорость ЕСЛИ качество ок | 10 мин pull + bench8 | отложено, жду user |
| 2 | Speculative decoding | +30–50% скорости | 1–2 дня миграции на llama.cpp | план готов, см. §10 |
| 3 | KTransformers | >2× ускорение | Нужен GPU + AVX-512 | при апгрейде железа |
| 4 | Intel AMX | 3–5× | Новый сервер | при обновлении парка |
| 5 | Sandwich | неизвестно | Wait for production | мониторить |
| 6 | T-SAR / BitNet | 10× | Нет русских моделей | 2027+ |
| 7 | SAIL / PIM | 20× | Новая память | не в этом десятилетии |

## 10. План миграции на llama.cpp

*(Документируется на случай будущего v1.6.0)*

### Инфраструктура

| Компонент | Что | Размер |
|---|---|---|
| `llama-server` | Бинарник из llama.cpp, замена Ollama | ~30 МБ |
| Target GGUF | `T-lite-it-2.1-Q4_K_M.gguf` (из Ollama blobs через симлинк) | ~5 ГБ |
| Draft GGUF | `Qwen3-0.6B-Instruct-Q8_0.gguf` | ~700 МБ |
| Итого RAM | Target + draft + KV-кэш обоих | ~7 ГБ |

**Критично:** draft должен иметь **тот же токенайзер**, что и target. Для T-lite (Qwen3-8B base) подходят только Qwen3-семейные мелкие: `Qwen3-0.6B-Base` или `Qwen3-0.6B-Instruct`. `qwen2.5:0.5b` НЕ подойдёт — другой токенайзер, acceptance rate упадёт к 10%.

### Шаги

```bash
# 1. llama.cpp
cd ~
wget https://github.com/ggml-org/llama.cpp/releases/latest/download/llama-bin-linux-avx2-x64.zip
unzip llama-bin-linux-avx2-x64.zip -d llama.cpp
chmod +x ~/llama.cpp/bin/llama-server

# 2. GGUF-файлы
find /usr/share/ollama/.ollama/models/blobs -name "sha256-*" -size +4G -size -6G
mkdir -p /home/service/llama/models
ln -s /usr/share/ollama/.ollama/models/blobs/sha256-<hash> \
      /home/service/llama/models/t-lite-2.1-q4.gguf
cd /home/service/llama/models
wget https://huggingface.co/Qwen/Qwen3-0.6B-Instruct-GGUF/resolve/main/Qwen3-0.6B-Instruct-Q8_0.gguf

# 3. Запуск с speculative decoding
numactl --cpunodebind=0 --membind=0 \
  ~/llama.cpp/bin/llama-server \
    --model      /home/service/llama/models/t-lite-2.1-q4.gguf \
    --model-draft /home/service/llama/models/Qwen3-0.6B-Instruct-Q8_0.gguf \
    --draft-max 7 --draft-min 1 --draft-p-min 0.75 \
    --threads 16 --threads-draft 4 \
    --ctx-size 4096 --ctx-size-draft 4096 \
    --flash-attn --host 0.0.0.0 --port 8081
```

### Правки в репо (минимальный PR v1.6.0)

| Файл | Изменение |
|---|---|
| `Сервер/local/main.py` | `call_ollama()` → `call_llama_server()`, URL `/api/chat` → `/v1/chat/completions` (OpenAI-compat), убрать `keep_alive` |
| `Сервер/local/.env.example` | `OLLAMA_URL` → `LLAMA_URL=http://localhost:8081`, `MODEL_PATH`, `DRAFT_MODEL_PATH` |
| `start.sh`/`start.bat` | Запуск llama-server перед FastAPI + ожидание `/health` |
| `/etc/systemd/system/llama-server.service` | Новый systemd-unit (рядом с ollama.service) |
| `Инструкции/LOCAL_MODEL.md` | Секция «Запуск на llama.cpp со speculative decoding» |
| `tests/test_servers_smoke.py` | Моки на OpenAI-compat endpoint |

### Риски

1. **Broadwell без AVX-512.** Архитектурного преимущества llama.cpp над Ollama здесь нет — ускорение **только** от speculative. Если Broadwell держится за 44 с на T-lite, speculative довезёт до ~28–30 с.
2. **Acceptance rate может быть 20–30%, не 50%.** Русский деловой стиль — узкий, draft 0.6B может плохо предсказывать. Тогда overhead съест выгоду. **До PR обязательно замер**: если acceptance <30% — миграция не нужна.
3. **Нет `keep_alive` API.** Модель всегда в RAM пока процесс жив. Плюс — нет cold-start, минус — постоянно занято ~7 ГБ.
4. **Ollama остаётся как fallback.** В `main.py` добавится ENV-флаг `LLAMA_BACKEND=llama-cpp|ollama`. Если что-то ломается — одна строка в `.env` откатывает.

### Когда делать

После того, как v1.5.0 уверенно поработает 2–3 дня на боевом и T-lite зарекомендует себя на реальных документах. Без этой базы смысла в +30% нет.

## 12. Деплой на прод: systemd-ловушки

### Ошибка 1: status=200/CHDIR

```
апр 29 18:17:08 ext (python3)[634067]: ai-suggester.service: Failed at step CHDIR
  spawning /usr/bin/python3: No such file or directory
апр 29 18:17:08 ext systemd[1]: ai-suggester.service: Main process exited,
  code=exited, status=200/CHDIR
```

**Диагноз:** `WorkingDirectory=/opt/ai-suggester` в unit-файле — это **плейсхолдер** из шаблона репозитория. На вашем сервере приложение лежит в `/home/service/llama/server/local/`. systemd пытается сделать `chdir("/opt/ai-suggester")`, папка не существует → status=200/CHDIR.

Плюс `User=YOUR_USERNAME` — тоже плейсхолдер.

**Исправление:**

```bash
sudo systemctl edit --full ai-suggester
# Привести к:
```

```ini
[Unit]
Description=AI LibreOffice Suggester — Local Server
After=network.target ollama.service
Requires=ollama.service

[Service]
Type=simple
User=service
Group=service
WorkingDirectory=/home/service/llama/server/local
EnvironmentFile=/home/service/llama/server/local/.env
ExecStart=/home/service/llama/server/local/venv/bin/python3 \
  -m uvicorn main:app --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### Ошибка 2: status=203/EXEC

```
апр 29 18:21:39 ext (python3)[634171]: Failed to locate executable
  /home/service/llama/server/local/.venv/bin/python3: No such file or directory
```

**Диагноз:** указал путь с точкой (`.venv`), а у пользователя venv назван без точки (`venv/` — видно из prompt'а `(venv) service@ext:~/llama/server/local$`).

**Исправление:**

```diff
-ExecStart=/home/service/llama/server/local/.venv/bin/python3 \
+ExecStart=/home/service/llama/server/local/venv/bin/python3 \
   -m uvicorn main:app --host 0.0.0.0 --port 8000
```

После правки:
```bash
sudo systemctl daemon-reload
sudo systemctl reset-failed ai-suggester  # сбросить 210+ auto-restart счётчик
sudo systemctl start ai-suggester
sudo systemctl status ai-suggester --no-pager
curl http://localhost:8000/health
```

### Урок для следующих раз

Шаблон `Сервер/local/ai-suggester.service` в репозитории содержит плейсхолдеры:
- `User=YOUR_USERNAME`
- `WorkingDirectory=/opt/ai-suggester`
- `ExecStart=/usr/bin/python3 ...`

Админ должен **обязательно** подменить их под свою инфраструктуру. Ни один из трёх нельзя оставить как есть.

Добавить в `ADMIN_GUIDE.md` чек-лист замен плейсхолдеров — отдельный TODO.
